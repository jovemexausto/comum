# Framing 2 — **Public Memory + Auditable Local Authority**
**Título provisório:** *Comum: Public Memory and Auditable Local Authority Without a Mandatory Center*

## 1. Hipótese principal
Infraestruturas de coordenação ficam politicamente e epistemicamente mais robustas quando tratam memória pública e autoridade local auditável como primitivas centrais, em vez de esconder autoridade sob neutralidade técnica.

## 2. Afirmação falsificável
Se o Comum não oferecer mais legibilidade de delegação, contestação, revogação e responsabilização do que arranjos locais informais ou plataformas centralizadas comparáveis, esse framing falha.

- melhor reconstrução ex post de decisões/disputas

## 4. Metodologia sugerida
- Design:
  - case study formalizado
  - scenario analysis
  - eventualmente pequeno piloto
- Métricas:
  - taxa de reconstrução correta de eventos

  - tempo para identificar autoridade relevante
- Viés:

I think this is the right move, with one important refinement:

**code and paper should be projections of corpus, but not direct copies of it.**

If `corpus` becomes the place of reasons, orientation, ontology, political thesis, and conceptual constraints, then:

- **paper** is an outward-facing argumentative projection of corpus + evidence
- **spec** is a normative contraction of corpus into interoperable commitments
- **code** is an executable projection of spec + implementation constraints
- **project** is the temporal/operational projection of what is being done now

So I’d frame it like this:

## Core thesis
`docs/corpus/` should be the source of:
- why the Comum exists
- what kind of world it is trying to enable
- what it refuses
- what its concepts mean

But it should **not** become:
- technical SSOT
- normative contract
- implementation handbook
- paper draft dump

## Artifact discipline
I’d define the artifacts by question:

- **Corpus**
  - answers: `why?`, `what is it?`, `what does it mean?`
- **Spec**
  - answers: `what must hold?`
- **Code**
  - answers: `how does it run?`
- **Evidence**
  - answers: `what has been demonstrated?`
- **Paper**
  - answers: `what can we responsibly claim to others now?`
- **Project**
  - answers: `what are we doing next?`
- **Notes**
  - answers: `what is still open, excess, or not yet integrated?`

## Best dependency graph
I’d make the discipline explicit as:

```text
notes -> corpus -> spec -> code
                 -> paper
code + tests + simulations -> evidence -> paper
project coordinates the timing of all of the above
```

More precisely:

- `notes` feed `corpus`
- `corpus` constrains `spec`
- `spec` constrains `code`
- `code`, tests, conformance, simulations produce `evidence`
- `paper` is written from `corpus + evidence`, not from aspiration alone

That last line is crucial.

## Strong rule
I would adopt this:

> The corpus is the source of orientation, not the source of bytes.

That prevents a dangerous confusion:
- corpus should shape meaning
- spec should shape contract
- code should shape execution

## What this buys you
1. **No more theory/code confusion**
- code is no longer asked to carry philosophy
- corpus is no longer asked to behave like a spec

2. **Paper becomes disciplined**
- not “whatever sounds good”
- but: corpus-filtered, evidence-bounded claims

3. **Spec becomes cleaner**
- it inherits concepts from corpus
- but only keeps what implementations must obey

4. **Notes stop competing**
- they become pre-corpus, pre-paper, pre-spec material

## Risks
There are 3 risks to avoid:

### 1. Corpus becoming too sovereign
If everything must “come from corpus,” the corpus can become a soft center.
So I’d say:

- corpus has **interpretive priority**
- spec has **normative priority**
- code has **operational priority**
- evidence has **epistemic priority** for public claims

### 2. Paper skipping evidence
A paper should never be:
- corpus rewritten for a venue

It must be:
- corpus filtered by what is already evidenced

### 3. Code skipping corpus
If code evolves without corpus, drift returns:
- technically coherent
- conceptually incoherent

### 1. `Corpus Artifact`

Must contain:
- thesis
