## 2. Afirmação falsificável

I think this is the right move, with one important refinement:

**code and paper should be projections of corpus, but not direct copies of it.**

If `corpus` becomes the place of reasons, orientation, ontology, political thesis, and conceptual constraints, then:

- **paper** is an outward-facing argumentative projection of corpus + evidence
- **spec** is a normative contraction of corpus into interoperable commitments
- **code** is an executable projection of spec + implementation constraints
- **project** is the temporal/operational projection of what is being done now

So I’d frame it like this:

## Core thesis
`docs/corpus/` should be the source of:
- why the Comum exists
- what kind of world it is trying to enable
- what it refuses
- what its concepts mean

But it should **not** become:
- technical SSOT
- normative contract
- implementation handbook
- paper draft dump

## Artifact discipline
I’d define the artifacts by question:

- **Corpus**
  - answers: `why?`, `what is it?`, `what does it mean?`
- **Spec**
  - answers: `what must hold?`
- **Code**
  - answers: `how does it run?`
- **Evidence**
  - answers: `what has been demonstrated?`
- **Paper**
  - answers: `what can we responsibly claim to others now?`
- **Project**
  - answers: `what are we doing next?`
- **Notes**
  - answers: `what is still open, excess, or not yet integrated?`

## Best dependency graph
I’d make the discipline explicit as:

```text
notes -> corpus -> spec -> code
                 -> paper
code + tests + simulations -> evidence -> paper
project coordinates the timing of all of the above
```

More precisely:

- `notes` feed `corpus`
- `corpus` constrains `spec`
- `spec` constrains `code`
- `code`, tests, conformance, simulations produce `evidence`
- `paper` is written from `corpus + evidence`, not from aspiration alone

That last line is crucial.

## Strong rule
I would adopt this:

> The corpus is the source of orientation, not the source of bytes.

That prevents a dangerous confusion:
- corpus should shape meaning
- spec should shape contract
- code should shape execution

## What this buys you
1. **No more theory/code confusion**
- code is no longer asked to carry philosophy
- corpus is no longer asked to behave like a spec

2. **Paper becomes disciplined**
- not “whatever sounds good”
- but: corpus-filtered, evidence-bounded claims

3. **Spec becomes cleaner**
- it inherits concepts from corpus
- but only keeps what implementations must obey

4. **Notes stop competing**
- they become pre-corpus, pre-paper, pre-spec material

## Risks
There are 3 risks to avoid:

### 1. Corpus becoming too sovereign
If everything must “come from corpus,” the corpus can become a soft center.
So I’d say:

- corpus has **interpretive priority**
- spec has **normative priority**
- code has **operational priority**
- evidence has **epistemic priority** for public claims

### 2. Paper skipping evidence
A paper should never be:
- corpus rewritten for a venue

It must be:
- corpus filtered by what is already evidenced

### 3. Code skipping corpus
If code evolves without corpus, drift returns:
- technically coherent
- conceptually incoherent

### 1. `Corpus Artifact`

- thesis

As diferenças são estas.

**1. Mesmo DID em tudo**
Você usa a mesma identidade operacional em todos os apps, comunidades e fluxos.

**Vantagens**
- simples de entender
- simples de implementar
- reputação e histórico ficam imediatamente acumulados
- menos atrito para portabilidade entre apps

**Custos**
- correlação total entre contextos
- um app ruim passa a observar mais da sua vida relacional
- pseudonimato fica fraco
- comprometimento de chave tem raio maior
- mais difícil separar papéis da mesma pessoa

**Quando faz sentido**
- comunidades pequenas
- contextos de alta confiança
- fase inicial de implementação
- casos em que reputação contínua é mais importante que privacidade contextual

---

**2. Raiz + personas**
Você tem uma identidade raiz soberana, mas opera com chaves/personas derivadas por app, comunidade ou contexto. Quando quiser, prova que elas pertencem à mesma pessoa.

**Vantagens**
- muito melhor privacidade
- menos correlação entre apps e comunidades
- permite separar papéis
- comprometimento de uma persona não destrói tudo
- combina melhor com ZK, commitments e provas seletivas

**Custos**
- arquitetura mais complexa
- UX mais delicada
- reputação não se soma automaticamente
- precisa de mecanismo explícito para “provar vínculo” entre personas
- comunidades precisam decidir quando aceitam persona isolada e quando exigem continuidade

**Quando faz sentido**
- longo prazo
- federação entre comunidades diferentes
- múltiplos apps sobre a mesma infraestrutura
- cenários com atributos privados, provas de idade, elegibilidade, reputação parcial

---

**3. O ponto decisivo**
A diferença real é:

- `mesmo DID em tudo` privilegia **continuidade e simplicidade**
- `raiz + personas` privilegia **soberania, privacidade e separação de contextos**

Pelo que você descreveu, eu recomendo **raiz + personas**.

Porque você quer ao mesmo tempo:
- desacoplar identidade do app
- usar a mesma pessoa em vários apps
- manter autonomia
- suportar metadados opcionais
- permitir provas sem expor dado bruto
- possibilitar atualização segura

Isso fica mais consistente assim:

- a pessoa controla uma **identidade raiz**
- cada app usa uma **persona operacional**
- atributos ficam ligados à raiz ou a commitments controlados pela raiz

- Se uma comunidade exigir prova de maioridade, você apresenta:

  Ex.: nascimento, maioridade, residência, filiação, licença
- **predicados derivados**
  Ex.: `18+`, `21+`, `mora_na_regiao_X`

- o fato bruto pode ficar privado
- o que circula é o predicado ou a prova
- a validade é avaliada por `issued_at`, `valid_from`, `expires_at`, revogação e contexto
- não depende de “editar para baixo ou para cima” um campo de perfil

Ou seja:
- não pense em “idade como metadata editável”
- pense em “credenciais e provas com escopo, tempo e revogação”

- um contrato assinado com prova válida naquele momento continua referenciando aquela prova/contexto
