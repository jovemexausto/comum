## Resumo (Abstract)

O Comum Protocol especifica uma infraestrutura minima de confianca para
comunidades que precisam se identificar, transacionar e se organizar sem
depender de intermediarios nao escolhidos. Ele define um unico primitivo,
o **Testemunho**, do qual derivam por composicao identidade autonoma,
reputacao emergente, transacoes privadas, contratos locais e emissao de
valor comunitario.

Nao existe servidor central, autoridade emissora, nem consenso global.
O estado de uma comunidade e uma propriedade emergente de um grafo de
Testemunhos distribuido entre os dispositivos dos participantes. A
convergencia e garantida por propriedades algebricas do modelo de dados,
nao por coordenacao politica.

O protocolo e offline-first e transport-agnostic. Ele funciona em internet
publica, BLE, NFC, QR fisico e malhas sem infraestrutura como Reticulum,
com um contrato minimo de transporte que permite operar ate em meios com
5 bps e MTU de 500 bytes.

## 1. Motivacao Politica

Todo sistema digital contemporaneo concentra tres poderes sobre cada pessoa:
o poder de emitir identidade, o poder de autorizar transacoes e o poder de
definir as regras de participacao. Esses poderes pertencem hoje a Estados,
corporacoes e plataformas, nao porque seja inevitavel, mas porque as
ferramentas alternativas ainda nao existiam.

Design e ideologia. Toda tecnologia codifica uma teoria sobre quem pode
coordenar quem. O Comum Protocol codifica uma teoria especifica: pessoas
devem poder se organizar entre si sem pedir permissao a intermediarios que
nao escolheram. Essa posicao e um principio de subsidiariedade tecnica: o
poder de coordenacao deve residir no nivel mais proximo possivel de quem e
afetado por ele.

O protocolo nao propõe abolir estruturas existentes. Ele cria a possibilidade
de comunidades que precisam, ou que escolhem, organizar-se fora delas com a
mesma solidez criptografica disponivel nos sistemas formais. E uma ferramenta
para autodeterminacao, nao um manifesto de ruptura.

## 2. Fundamentos Teoricos

### 2.1 Confianca distribuida

O problema central da coordenacao sem autoridade central e: como dois agentes
que nunca se encontraram podem transacionar com seguranca sem recorrer a um
terceiro ja confiavel? A resposta historica e a transitividade da confianca:
se Alice confia em Bob e Bob confia em Carlos, Alice pode atribuir algum grau
de confianca a Carlos, proporcional a confianca em Bob e a reputacao de Bob
como intermediario confiavel.

O Comum Protocol formaliza essa intuicao em estrutura de dados verificavel e
composicional.

### 2.2 O Testemunho como primitivo minimo

Um **Testemunho** e a menor unidade de informacao verificavel que um agente
pode produzir sobre um evento no mundo. E uma afirmacao assinada de que algo
aconteceu, por quem estava presente, no momento em que aconteceu.

### 2.3 O grafo como estado global

Testemunhos se referenciam por hash criptografico, formando um grafo dirigido
aciclíco (DAG). A referencia codifica causalidade: B veio depois de A, B sabe
de A, B responde a A. O estado global de uma comunidade nao e armazenado em
lugar nenhum; e uma propriedade emergente do grafo.

### 2.4 Convergencia sem coordenacao

Para que o estado derivado do grafo seja consistente entre nos com subgrafos
diferentes, o modelo precisa satisfazer comutatividade, associatividade e
idempotencia. A operacao de merge definida como uniao de conjuntos satisfaz
essas propriedades. Portanto, qualquer no que receba qualquer subconjunto de
Testemunhos converge ao mesmo estado quando o grafo se completa.

## 3. O Testemunho

### 3.1 Estrutura formal

```
Testemunho {
  version:   u8
  id:        bytes[32]    // SHA3-256 de todos os campos exceto id
  author:    bytes[32]?   // opcional em transportes que ja codificam a origem
  timestamp: i64          // Unix ms (UTC)
  suite:     u16          // suite criptografica

  refs:      bytes[32][]  // referencias causais
  claim:     Claim        // afirmacao semantica
  proof:     Proof        // assinatura + provas ZK + nullifiers
}
```

`author` e opcional quando o transporte ja provê identidade como endereco
de origem (ex.: Reticulum). Quando ausente, implementacoes devem inferir o
autor a partir do transporte e considerar o Testemunho invalido se houver
inconsistencia.

### 3.2 Claim

```
Claim {
  verb:    string
  payload: bytes[]  // CBOR
}
```

O protocolo transporta e verifica; nao interpreta. Verbos sao definidos por
comunidades e Capsulas.

Verbos de referencia (recomendados):

| Verb | Semantica |
|------|-----------|
| `comum/transfer` | transferencia de valor entre dois DIDs |
| `comum/vote` | voto em proposta |
| `comum/propose` | submissao de proposta |
| `comum/encounter` | encontro fisico co-assinado |
| `comum/vouch` | endosso de reputacao |
| `comum/revoke` | revogacao social |
| `capsule/invoke` | invocacao de Capsula |
| `capsule/result` | resultado de Capsula |
| `genesis` | fundacao comunitaria |

### 3.3 Proof

```
Proof {
  signature:  bytes[]   // assinatura sobre id
  zk_proofs:  ZKProof[] // provas opcionais
  nullifiers: bytes[][] // anti double-spend
}
```

**Assinatura** garante autoria e integridade. Algoritmo depende da suite.
**ZK proofs** sao opcionais e especificas por verbo. **Nullifiers** impedem
gasto duplo em verbos de transferencia. O modelo simples e:

```
nullifier = PRF(sk, testemunho_id)
```

Limite conhecido: sem compromisso previo, a prova de nao gasto e local e
nao global. Esta versao prefere simplicidade para implementacao em
smartphones e explicita essa limitacao. Versoes futuras podem introduzir
commitments versionados sem breaking change no campo `nullifiers`.

## 4. Identidade

### 4.1 Nascimento por ato

Nao existe cadastro. Uma identidade nasce quando um dispositivo gera seu
primeiro par de chaves e assina seu primeiro Testemunho.

```
Inicializacao:
  1. Gerar (sk, pk) via CSPRNG
  2. DID = "did:comum:" + base58(SHA3-256(pk)[:20])
  3. sk em enclave seguro
  4. Assinar Testemunho inicial
```

O DID e um pseudonimo derivado da chave. Nao carrega nome, email ou
localizacao.

### 4.2 Reputacao emergente

Reputacao e calculada do grafo, nao armazenada. A funcao de reputacao e
definida pela Capsula de identidade da comunidade, nao pelo protocolo.

### 4.3 Revogacao social

Dispositivo comprometido: membros da rede de confianca emitem `comum/revoke`
co-assinando a revogacao. O limiar M-of-N e definido pela comunidade. Nao
existe autoridade central de revogacao.

## 5. Suites Criptograficas

### 5.1 Suites definidas

| ID | Nome | Assinatura | ZK | Hardware alvo | Status |
|----|------|-----------|-----|---------------|--------|
| `0x0001` | CLASSIC_1 | Ed25519 | Bulletproofs | smartphone, desktop | obrigatoria |
| `0x0002` | PQ_HYBRID_1 | Ed25519 + Falcon-512 | Bulletproofs | smartphone | recomendada |

Suite MINIMAL e suportada apenas como excecao em hardware extremamente
limitado e nao e considerada suite de primeira classe.

### 5.2 Negociacao por sessao

```
-> HELLO     { suites: [0x0001, 0x0002], pub_key: ... }
<- HELLO_ACK { suite:  0x0001, pub_key: ..., session_id: ... }
```

Suite escolhida: maior ID suportado por ambos.

## 6. Wallet e Comuns

### 6.1 Definicao

**Comuns** e a unidade de valor comunitario local. Nao existe Comum global.
Oferta e politica de emissao sao definidas pela Capsula de emissao da
comunidade.

### 6.2 Saldo como propriedade emergente

```
saldo(did, t) =
  sum(recebidos) - sum(enviados) + sum(mints)
```

Saldo nao e armazenado. E calculado do grafo.

### 6.3 Transferencia offline

```
Alice paga 5 Comuns para Bob via NFC:

1. Alice emite Testemunho:
   { verb: "comum/transfer",
     payload: { to: did_bob, amount: 5 },
     proof: { signature: Ed25519(sk_alice, id),
              zk_proofs: [ bulletproof("saldo >= 5") ],
              nullifiers: [ PRF(sk_alice, id) ] } }

2. Bob verifica: assinatura valida, ZK valida, nullifier novo
3. Bob emite recibo referenciando o Testemunho de Alice
```

### 6.4 Emissao por presenca sem GPS

GPS nao e requisito. Presenca e definida por uma **ProvaDeContexto** modular
interpretada pela Capsula de emissao.

```
ProvaDeContexto {
  tipo:    "proximidade" | "beacon" | "lugar" | "vouch" | "nenhum"
  payload: bytes[]
  proof:   Proof
}
```

Testemunho de encontro:

```
{ verb: "comum/encounter",
  payload: { context: ProvaDeContexto, nonce: random() },
  refs: [ id_contraparte ],
  proof: { signature_a, signature_b } }
```

Provas de contexto de referencia:
- proximidade: NFC/BLE como prova fisica
- beacon: token assinado pela comunidade em um ponto fisico
- lugar: hash de locais conhecidos da comunidade
- vouch: co-assinatura de membros estabelecidos

## 7. Contratos — Capsulas

### 7.1 Definicao e modelo

Uma **Capsula** e um programa WASM deterministico que le Testemunhos e emite
novos Testemunhos. Sem servidor, sem blockchain, sem gas.

```
Capsula: (subgrafo, parametros) -> [Testemunho]
```

### 7.2 Syscalls

```rust
fn read(filter: Filter) -> Vec<Testemunho>;
fn sign(payload: &[u8]) -> Proof;
fn emit(claim: Claim, refs: Vec<Id>, proof: Proof) -> Id;
```

Nada mais. Capsulas nao acessam filesystem, rede, camera ou localizacao.

### 7.3 Manifesto

```
CapsuleManifest {
  id:        bytes[32]   // hash do WASM
  name:      string
  version:   string
  author:    DID
  permissions: ["read", "emit", "sign"]
}
```

### 7.4 Ciclo de vida

```
PROPOSTA -> VOTACAO -> APROVADA -> ATIVA -> DEPRECADA -> ARQUIVADA
             |
          REJEITADA
```

## 8. Transporte

### 8.1 Contrato minimo

Um transporte valido suporta half-duplex, MTU >= 500 bytes e throughput
medio >= 5 bps. Esse contrato e inspirado por Reticulum e permite operar em
meios extremos como radio de baixa taxa.

### 8.2 Transportes de referencia

| Transporte | Alcance | MTU | Uso |
|-----------|---------|-----|-----|
| HTTPS/WS | global | 64KB | internet disponivel |
| BLE 5.0 | ~10m | 512B | smartphones proximos |
| NFC | ~5cm | 8KB | transacao presencial |
| Reticulum | km+ | 500B+ | mesh sem infraestrutura |
| QR fisico | assincrono | ~3KB | papel, offline absoluto |

Reticulum e transporte de primeira classe. Seu mecanismo de announce serve
como descoberta sem servidor. Em Reticulum, `author` pode ser omitido.

### 8.3 Sincronizacao

```
-> SYNC_HELLO { vclock }
<- SYNC_DELTA { testemunhos_novos }
-> SYNC_DELTA { testemunhos_novos }
<- SYNC_ACK
```

### 8.4 Gossip

Testemunhos marcados como broadcast sao retransmitidos com TTL e deduplicacao
por Bloom Filter.

## 9. Armazenamento e Ciclo de Vida

### 9.1 O problema honesto

Sem poda, o grafo cresce indefinidamente. Sem snapshots, nos leves nao
conseguem validar historico.

### 9.2 Epoch snapshots

```
EpochSnapshot {
  epoch:      u32
  period:     (Timestamp, Timestamp)
  balances:   MerkleRoot
  reputation: MerkleRoot
  nullifiers: MerkleRoot
  capsules:   MerkleRoot
  prev:       EpochSnapshotId
  signatures: [Proof]
}
```

O conjunto de co-assinantes e politica de aprovacao sao definidos pela
Capsula de governanca da comunidade. O protocolo define apenas o formato e
a verificacao.

### 9.3 Perfis de no

| Perfil | Armazena |
|--------|---------|
| `LIGHT` | epoca corrente + snapshots recentes |
| `FULL` | ano corrente + snapshots |
| `ARCHIVE` | grafo completo desde o Genesis |

## 10. Genesis

Uma comunidade nasce por ato coletivo e cerimonia fisica. O Genesis cria a
chave comunitaria por threshold signature (ex.: FROST).

```
Testemunho de Genesis:
{ verb: "genesis",
  payload: { name, threshold, founders, capsules, supply, mint_policy },
  proof: { signatures: [founders] } }
```

## 11. Modelo de Ameacas

**Protege:** identidade civil, conteudo de transacoes, continuidade sem
internet, integridade do grafo.

**Nao protege por completo:**
- correlacao temporal por observacao fisica do espectro
- dispositivo comprometido fisicamente
- ataques Sybil em estagios iniciais (mitigados por reputacao emergente)

**Ameacas especificas de transporte:**
- BLE: exige MAC rotation por sessao
- Reticulum: analise de trafego em malhas densas ainda possivel
