## Resumo (Abstract)

O Comum Protocol é uma infraestrutura de confiança mínima para comunidades
que precisam coordenar, transacionar e governar sem delegar soberania a
terceiros. O protocolo define um único primitivo — o **Testemunho** — do qual
derivam, por composição, identidade autônoma, reputação emergente, transações
privadas, contratos locais e emissão de valor comunitário.

Não existe servidor central, autoridade emissora, nem consenso global. O
estado da rede é uma propriedade emergente do grafo de Testemunhos
distribuídos entre os dispositivos dos próprios participantes. Convergência
é garantida matematicamente por propriedades algébricas do modelo de dados,
não por coordenação política.

O protocolo funciona sobre qualquer meio de transporte — da internet pública
ao NFC entre dois dispositivos sem bateria — e é projetado para sobreviver à
obsolescência criptográfica, ao colapso de infraestrutura, e à hostilidade
de Estados.

---

## 1. Motivação Política

Todo sistema de confiança digital contemporâneo pressupõe uma autoridade
emissora: o Estado que concede o CPF, a empresa que valida o e-mail, a
plataforma que emite o token. Essa arquitetura não é acidental — é a
transposição do monopólio estatal sobre a identidade para o domínio digital.

O resultado prático é que participar da economia digital requer a permissão
de alguma entidade com poder de revogação. Quem não tem CPF não abre conta.
Quem não tem conta não acessa serviços. Quem incomoda o poder tem sua
identidade digital suspensa. A exclusão não é um bug — é uma característica
de arquitetura.

O Comum Protocol parte de uma premissa diferente, herdada da tradição
anarquista de organização horizontal: **a confiança emerge das relações entre
pessoas, não da certificação por autoridades.** Uma comunidade que se organiza,
transaciona e governa localmente não precisa de permissão externa para existir.
Precisa apenas de instrumentos que tornem essas relações verificáveis sem
exposição, persistentes sem servidor, e soberanas sem Estado.

Esses instrumentos são o que este documento especifica.

## 2. Fundamentos Teóricos

### 2.1 O problema da confiança distribuída

O problema central de qualquer sistema de coordenação sem autoridade central
é o seguinte: como dois agentes que nunca se encontraram podem estabelecer
confiança suficiente para transacionar, sem recorrer a um terceiro que os
dois já confiam?

A resposta clássica — desde as hawalas islâmicas do século VIII até o
web of trust do PGP — é a transitividade da confiança: se Alice confia em
Bob e Bob confia em Carlos, Alice pode atribuir algum grau de confiança a
Carlos, proporcional à confiança que deposita em Bob e à reputação de Bob
como intermediário confiável.

O Comum Protocol formaliza essa intuição em estrutura de dados verificável.

### 2.2 O Testemunho como primitivo

Um **Testemunho** é a menor unidade de informação verificável que um agente
pode produzir sobre um evento no mundo. É a afirmação assinada de que algo
aconteceu, por quem estava presente, no momento em que aconteceu.

Formalmente, um Testemunho é um objeto imutável com quatro campos:

```
Testemunho {
  quem:    DID           // identidade do agente que testemunha
  oque:    Intent        // afirmação semântica livre sobre o evento
  quando:  Timestamp     // momento da assinatura (relógio local)
  prova:   Signature     // prova criptográfica de autoria e integridade
}
```

A propriedade central do Testemunho é sua **imutabilidade por construção**:
a assinatura cobre todos os campos. Alterar qualquer campo invalida a prova.
Um Testemunho válido é uma afirmação que seu autor não pode negar
(non-repudiation) e ninguém pode forjar.

### 2.3 O grafo como estado global

Testemunhos se referenciam uns aos outros por hash criptográfico, formando
um grafo dirigido acíclico (DAG) — a mesma estrutura subjacente ao Git,
ao Bitcoin, e ao ActivityPub. A referência de B a A codifica causalidade:
B veio depois de A, B sabe de A, B responde a A.

```
Testemunho A <── referencia── Testemunho B <──referencia── Testemunho C
```

O estado global da rede — saldos, identidades, reputações, resultados de
votações — não é armazenado em lugar nenhum. É uma **propriedade emergente
do grafo**: calculada localmente por qualquer nó que possua o subgrafo
relevante, deterministicamente, sem coordenação.

Não existe banco de dados. Existe apenas o grafo, e o grafo é o banco de dados.

### 2.4 Convergência sem coordenação

Para que o estado derivado do grafo seja consistente entre nós que possuem
subgrafos diferentes, o modelo de dados deve satisfazer três propriedades:

```
merge(A, B) = merge(B, A)                     // comutatividade
merge(A, merge(B, C)) = merge(merge(A, B), C) // associatividade
merge(A, A) = A                               // idempotência
```

Um conjunto de Testemunhos com operação de merge definida como união satisfaz
trivialmente essas três propriedades. Não existe conflito possível: dois
Testemunhos diferentes são dois fatos diferentes. A rede converge ao estado
correto quando qualquer nó recebe qualquer subconjunto dos Testemunhos
existentes, em qualquer ordem.

Isso é o que a literatura de sistemas distribuídos chama de strong eventual
consistency (Shapiro et al., 2011) — a garantia certa para uma rede
comunitária offline-first.

## 3. Especificação do Primitivo

### 3.1 Estrutura completa

```
Testemunho {
  // Cabeçalho — 83 bytes fixos
  version:   u8           // versão do protocolo (atualmente: 2)
  id:        bytes[32]    // SHA3-256(todos os outros campos)
  author:    bytes[32]    // DID do autor (hash da chave pública)
  timestamp: i64          // Unix timestamp em milissegundos (UTC)
  suite:     u16          // suite criptográfica (ver Seção 5)

  // Referências causais
  refs:      bytes[32][]  // hashes de Testemunhos anteriores

  // Intenção
  intent:    Intent       // afirmação semântica (ver 3.2)

  // Prova
  proof:     Proof        // autoria + claims ZK + nullifiers (ver 3.3)
}
```

O campo `id` é derivado do conteúdo — nunca atribuído externamente.
Isso garante content-addressability: dois Testemunhos com o mesmo conteúdo
produzem o mesmo `id`, e qualquer modificação produz um `id` diferente.

Tamanho mínimo: ~120 bytes. Tamanho máximo: 64KB.

### 3.2 Intent — a afirmação semântica

```
Intent {
  verb:    string    // ação em namespace hierárquico livre
  payload: bytes[]   // dados específicos do verb, serializado em CBOR
}
```

O protocolo não define verbos obrigatórios. Verbos são definidos por
comunidades e Cápsulas. O protocolo transporta e verifica — não interpreta.

Verbos de referência (recomendados, não obrigatórios):

| Verb              | Semântica                                       |
| ----------------- | ----------------------------------------------- |
| `comum/transfer`  | transferência de valor entre dois DIDs          |
| `comum/vote`      | voto em proposta identificada por hash          |
| `comum/propose`   | submissão de proposta para votação              |
| `comum/encounter` | encontro físico co-assinado por ambas as partes |
| `comum/vouch`     | endosso de reputação de outro DID               |
| `comum/revoke`    | revogação de Testemunho anterior próprio        |
| `capsule/invoke`  | invocação de Cápsula com parâmetros             |
| `capsule/result`  | resultado de execução de Cápsula                |
| `genesis`         | fundação de nova comunidade                     |

### 3.3 Proof — a prova criptográfica

```
Proof {
  signature:  bytes[]   // assinatura digital sobre o id do Testemunho
  zk_proofs:  ZKProof[] // provas ZK opcionais sobre claims do payload
  nullifiers: bytes[][] // anti double-spend para verbos de gasto
}
```

**Assinatura** — garante autoria e integridade. Algoritmo determinado pela
suite. Para CLASSIC_1: Ed25519, 64 bytes.

**ZK Proofs** — provas de conhecimento zero sobre afirmações no payload.
Presença opcional — apenas verbos que requerem privacidade as incluem.
Para CLASSIC_1: Bulletproofs (range proofs compactos, <1KB).

**Nullifiers** — valores derivados deterministicamente da chave privada:

```
nullifier = PRF(chave_privada, testemunho_id)
```

Obrigatório em verbos de transferência. Um nó que recebe um Testemunho de
gasto verifica que o nullifier não existe no conjunto conhecido antes de
aceitar. Equivalente funcional ao UTXO do Bitcoin — sem ledger global.

## 4. Identidade

### 4.1 Nascimento por ato

Não existe cadastro. Uma identidade no Comum Protocol nasce no momento em
que um dispositivo gera seu primeiro par de chaves e assina seu primeiro
Testemunho. O ato de testemunhar é o ato de existir.

```
Inicialização:
  1. Gerar (sk, pk) via CSPRNG do hardware
  2. DID = "did:comum:" + base58check(SHA3-256(pk)[:20])
  3. sk → enclave seguro (nunca sai do hardware)
  4. Assinar primeiro Testemunho: { verb: "genesis/identity", payload: {pk} }
  5. Existir.
```

O DID é o único identificador público. Não carrega nome, CPF, e-mail, nem
localização. É um pseudônimo derivado da chave — verificável por qualquer um,
vinculável a uma identidade civil por ninguém, exceto o próprio portador,
por ato voluntário e explícito.

### 4.2 Reputação emergente

Reputação não é armazenada. É calculada do grafo:

```
reputação(did, comunidade) = f(
  { t ∈ grafo | t.author = did  ∨  did ∈ refs_transitivos(t) }
)
```

A função `f` é definida pela Cápsula de identidade da comunidade — não pelo
protocolo. Uma comunidade pode ponderar transações honradas, votos
participados, endossos de membros estabelecidos, ou qualquer combinação.

Esta separação é intencional: permitir que o protocolo defina reputação seria
introduzir pela porta dos fundos a autoridade central que ele se propõe a
eliminar.

### 4.3 Revogação social

Dispositivo comprometido: membros da rede de confiança do portador emitem
Testemunhos `comum/revoke` co-assinando a revogação. Quando M-of-N contatos
confiáveis co-assinam, implementações consideram o DID suspenso. O limiar
M-of-N é configurado pela comunidade. Não existe autoridade de revogação
central.

O processo é lento — propositalmente. Sistemas que revogam identidades
rapidamente e unilateralmente são sistemas que podem ser weaponizados.

## 5. Suites Criptográficas

### 5.1 Suites definidas

| ID       | Nome        | Assinatura           | ZK           | Hardware alvo       | Status      |
| -------- | ----------- | -------------------- | ------------ | ------------------- | ----------- |
| `0x0001` | CLASSIC_1   | Ed25519              | Bulletproofs | smartphone, desktop | obrigatória |
| `0x0002` | PQ_HYBRID_1 | Ed25519 + Falcon-512 | Bulletproofs | smartphone          | recomendada |
| `0x0003` | PQ_1        | Falcon-512           | Bulletproofs | smartphone          | futura      |
| `0x00FF` | MINIMAL     | Ed25519              | —            | nRF52840            | fallback    |

**Falcon-512 em vez de Dilithium3:** sobre LoRa com MTU de 255 bytes, uma
assinatura Dilithium3 (3.293 bytes) exige 13+ fragmentos com probabilidade
de entrega composta < 70% em mesh típico. Falcon-512 (666 bytes) exige 3-4
fragmentos com probabilidade > 97%. O custo computacional adicional (~300ms
no Cortex-M4F vs ~105ms) é trade-off favorável. (NIST FIPS 206, 2024.)

### 5.2 Negociação por sessão

```
→ HELLO     { suites: [0x0001, 0x0002], pub_key: ... }
← HELLO_ACK { suite:  0x0001, pub_key: ..., session_id: ... }
```

Suite escolhida: maior ID suportado por ambos. A heterogeneidade é tolerada —
um nó MINIMAL pode se comunicar com um nó PQ_1 usando CLASSIC_1.

## 6. Wallet e Comuns

### 6.1 O que são Comuns

**Comuns** é a unidade de valor comunitário local. Não é uma moeda
especulativa. É um instrumento de coordenação econômica análogo às moedas
complementares dos sistemas LETS, com privacidade criptográfica e operação
offline.

Cada comunidade emite seus próprios Comuns. Não existe Comum global. A oferta
total e a política de emissão são definidas pela Cápsula de emissão, aprovada
pela própria comunidade por votação.

### 6.2 Saldo derivado do grafo

```
saldo(did, t) =
  Σ { amount | T ∈ grafo, verb="comum/transfer", T.to=did, T.t≤t, nullifier válido }
− Σ { amount | T ∈ grafo, verb="comum/transfer", T.from=did, T.t≤t, nullifier válido }
+ Σ { amount | T ∈ grafo, verb="comum/mint", T.to=did, T.t≤t, aprovado por Cápsula }
```

Saldo não é armazenado em lugar nenhum. É calculado do grafo por qualquer nó
com o subgrafo relevante. O grafo é o ledger.

### 6.3 Transferência offline

```
Alice paga 5 Comuns para Bob via NFC:

1. Alice constrói Testemunho:
   { verb:    "comum/transfer",
     payload: { to: did_bob, amount: 5 },
     proof:   { signature:  Ed25519(sk_alice, id),
                zk_proofs:  [ bulletproof("saldo ≥ 5") ],
                nullifiers: [ PRF(sk_alice, id) ] } }

2. Transmissão NFC — ~800 bytes, ~3 segundos

3. Bob verifica: assinatura válida · ZK válida · nullifier novo → aceita

4. Bob emite recibo referenciando o Testemunho de Alice:
   { verb: "comum/receive", refs: [id_alice] }

5. Na próxima sincronização com qualquer nó:
   ambos os Testemunhos propagam pela rede
   nullifier de Alice existe agora em centenas de nós
   double-spend impossível
```

Dados revelados a terceiros: zero.
Dados revelados a Bob: que Alice tem saldo ≥ 5 (não o saldo exato).

### 6.4 Emissão — Proof of Presence

Comuns são emitidos por participação comunitária verificável. O mecanismo
de referência é o encontro físico co-assinado:

```
Testemunho de encontro:
{ verb:    "comum/encounter",
  payload: { event: hash_evento, locality: hash(gps ÷ 500m) },
  refs:    [ id_testemunho_contraparte ],
  proof:   { signature_a: Ed25519(sk_alice, id),
             signature_b: Ed25519(sk_bob, id) } }  // consentimento mútuo
```

O encontro sozinho não emite nada. A Cápsula de emissão da comunidade lê
o grafo de encontros do período e distribui Comuns conforme política aprovada
pela comunidade.

Proteções contra abuso:
- **Cooling period por par:** mesmo par produz encounters com valor decrescente
  geometricamente após o primeiro por período (padrão: 24h). Farming inviável.
- **Contexto obrigatório:** apenas encontros dentro de eventos reconhecidos
  pela Cápsula ativam emissão.
- **Teto configurável:** oferta máxima de Comuns requer votação para alterar.

## 7. Contratos — Cápsulas

### 7.1 Definição

Uma **Cápsula** é um programa WASM determinístico que lê Testemunhos do grafo
e emite novos Testemunhos como resultado. É um contrato inteligente que roda
localmente, sem servidor, sem blockchain, sem gas fees.

```
Cápsula: (subgrafo_relevante, parâmetros) → [Testemunho]
```

O determinismo é o contrato central: dado o mesmo subgrafo, a Cápsula produz
sempre os mesmos Testemunhos de saída, em qualquer dispositivo, em qualquer
momento. Qualquer nó pode verificar uma execução sem re-executá-la — apenas
verificando o hash do resultado contra o subgrafo de entrada.

### 7.2 Três syscalls

```rust
fn read(filter: Filter) -> Vec<Testemunho>;   // ler grafo local
fn sign(payload: &[u8]) -> Proof;             // assinar via enclave
fn emit(intent: Intent, refs: Vec<Id>, proof: Proof) -> Id; // emitir
```

Nada mais. Uma Cápsula não pode fazer requisições HTTP, ler o sistema de
arquivos, ou acessar câmera, microfone, ou localização precisa. A sandbox é
o contrato social com os usuários da Cápsula.

### 7.3 Ciclo de vida

```
PROPOSTA → VOTAÇÃO → APROVADA → ATIVA → DEPRECADA → ARQUIVADA
               ↓
           REJEITADA
```

Uma Cápsula não roda sem aprovação por votação da comunidade. O processo de
aprovação é ele mesmo uma Cápsula — a Cápsula de governança, presente desde
o Genesis.

### 7.4 Exemplos canônicos

**Votação:**
```
Lê:     Testemunhos { verb: "comum/vote", proposal: X } no período
Verifica: membro válido (ZK proof) · voto único (nullifier)
Emite:  { verb: "comum/vote_result", winner, tally }
        quando prazo encerra ou quórum é atingido
```

**Aprovação de gasto:**
```
Lê:     proposta + votos sobre ela
Verifica: quórum · dentro do teto orçamentário
Emite:  { verb: "comum/approved_transfer", amount, to, reason }
        → Cápsula de wallet executa automaticamente
```

**Minting por presença:**
```
Acorda: fim do período de emissão (timer semanal)
Lê:    encounters do período, agrupados por DID
Calcula: participação ponderada com cooling period
Emite:  N × { verb: "comum/mint", to, amount }
```

## 8. Transporte

### 8.1 Opacidade

O protocolo não sabe qual transporte está sendo usado. Interface mínima:

```
send(bytes, hint) -> Result
receive() -> (bytes, hint)
broadcast(bytes) -> Result
```

Um Testemunho serializado em CBOR é bytes. O transporte entrega bytes.

### 8.2 Transportes de referência

| Transporte  | Alcance    | MTU     | Uso                                    |
| ----------- | ---------- | ------- | -------------------------------------- |
| HTTPS/WS    | global     | 64KB    | protótipo, internet disponível         |
| BLE 5.0     | ~10m       | 512B    | smartphones próximos                   |
| NFC         | ~5cm       | 8KB     | transação presencial de alta confiança |
| LoRa 915MHz | ~km        | 51–255B | mesh sem infraestrutura                |
| QR físico   | assíncrono | ~3KB    | papel, offline absoluto                |

BLE: MAC rotation obrigatória a cada sessão — impossibilita rastreamento
por terceiros monitorando o espectro.

LoRa com suites PQ: usar erasure coding para fragmentos — tolerância a
perda sem retransmissão.

### 8.3 Sincronização oportunística

```
→ SYNC_HELLO { vetor_de_versão: VClock }
← SYNC_DELTA { testemunhos_novos: [Testemunho] }
→ SYNC_DELTA { testemunhos_novos: [Testemunho] }
← SYNC_ACK
```

Apenas os Testemunhos ausentes no peer são transmitidos. Em sincronizações
frequentes, o delta é tipicamente zero. Um dispositivo offline por 30 dias
sincroniza completamente ao primeiro contato com qualquer nó.

### 8.4 Gossip em mesh

Testemunhos com `broadcast: true` são retransmitidos por nós intermediários.
TTL decrementado a cada hop (padrão inicial: 7). Bloom Filter de ids vistos
(janela: 72h) evita loops.

## 9. Armazenamento

### 9.1 Epoch snapshots

A cada 30 dias (configurável), a comunidade produz um snapshot coletivo:

```
EpochSnapshot {
  epoch:      u32
  period:     (Timestamp, Timestamp)
  balances:   MerkleRoot
  reputation: MerkleRoot
  nullifiers: MerkleRoot
  capsules:   MerkleRoot
  prev:       EpochSnapshotId
  signatures: [Proof]          // M-of-N membros co-assinam
}
```

Após aprovação, Testemunhos anteriores ao período podem ser descartados
por nós com pouca memória. O snapshot é prova suficiente do estado histórico.
Qualquer claim sobre o passado é provado por Merkle proof contra o snapshot.

### 9.2 Perfis de nó

| Perfil    | Armazena                                   |
| --------- | ------------------------------------------ |
| `LIGHT`   | época corrente + snapshots recentes (~1MB) |
| `FULL`    | ano corrente + todos snapshots (~500MB)    |
| `ARCHIVE` | grafo completo desde o Genesis             |

Declarado no HELLO. Peers sabem o que podem solicitar.

## 10. Genesis

Uma comunidade nasce por ato coletivo. Mínimo: 3 pessoas presentes fisicamente.

```
Processo:
  1. Cerimônia de geração de chave multi-party
     (nenhum fundador conhece a chave inteira)
  2. Chave comunitária: threshold signature M-of-N
  3. Fundadores co-assinam Testemunho de Genesis:
     { verb: "genesis",
       payload: { name, threshold, founders, capsules,
                  supply, mint_policy, locality } }
  4. Este Testemunho é o primeiro no grafo da comunidade
  5. A comunidade existe — sem permissão de ninguém
```

O Testemunho de Genesis é verificável por qualquer pessoa. Irrevogável por
qualquer indivíduo. Os fundadores podem se dissipar — a comunidade continua
enquanto seu grafo existir em pelo menos um dispositivo.

## 11. Modelo de Ameaças

**O que o protocolo protege:**
identidade civil dos participantes · conteúdo de transações ·
grafo de relacionamentos · continuidade operacional sem internet

**O que não protege, e por quê:**

*Análise de tráfego físico:* correlação de padrões de encontro por observação
do espectro. Mitigação: locality usa hash de coordenadas arredondadas para
500m. Em ambientes de alto risco, NFC (< 5cm) em vez de BLE (~10m).

*Dispositivo comprometido fisicamente:* extração de chave do enclave.
Mitigação: hardware EAL 6+ (OPTIGA Trust M). Revogação social descrita
na Seção 4.3.

*Ataques Sybil:* múltiplos DIDs falsos. Mitigação: DIDs novos têm reputação
zero. Reputação significativa requer histórico de participação física real.

*Computadores quânticos:* Ed25519 vulnerável ao algoritmo de Shor.
Mitigação: suite PQ_HYBRID_1 disponível hoje; migração gradual via
negociação de suite sem breaking change.

*Censura de transporte:* bloqueio de HTTPS por adversário que controla a rede.
Mitigação: protocolo funciona sobre BLE, NFC, LoRa, e QR físico.

## 13. Governança do Protocolo

CC0 1.0 Universal — domínio público. Nenhuma entidade possui o protocolo.

Mudanças propostas como CIPs (Comum Improvement Proposals) no repositório
público, modelo IETF. Qualquer pessoa propõe. Discussão pública. Consenso
rough é suficiente. Sem comitê, sem fundação, sem token de governança.

O protocolo governa a si mesmo da forma que especifica que comunidades devem
se governar: horizontalmente, por Testemunhos públicos e consenso emergente.

## Apêndice A — Transação completa (offline, NFC)

```
Alice paga 5 Comuns para Bob por um disco.
Ambos offline. Alice tem saldo 23. Nenhum terceiro presente.

1. Alice constrói:
   { verb: "comum/transfer",
     payload: { to: did_bob, amount: 5 },
     proof: { signature:  Ed25519(sk_alice, id),
              zk_proofs:  [ bulletproof("saldo ≥ 5") ],
              nullifiers: [ PRF(sk_alice, id) ] } }

2. NFC — ~800 bytes, ~3 segundos

3. Bob: assinatura válida · ZK válida · nullifier novo → aceita

4. Bob: { verb: "comum/receive", refs: [id_alice] }

5. NFC inverso — ACK

6. Na próxima quinta-feira, Bob toca BLE com alguém com internet.
   Em ~400ms, os dois Testemunhos propagam.
   Nullifier de Alice existe em centenas de nós.
   Double-spend impossível.

Bytes trocados:            ~1.400
Dados revelados:           zero
Tempo total:               ~6 segundos
Autoridade envolvida:      nenhuma
```

## Apêndice B — Por que não blockchain

Blockchain resolve o problema de **ordenação global de eventos em redes
sem confiança**. É uma solução engenhosa e cara: cada nó mantém toda a
história, todos concordam na ordem de cada evento, um mecanismo de consenso
impede reescrita.

O Comum Protocol não precisa de ordenação global. Uma transação entre Alice
e Bob é um fato local. A única ordem relevante é causalidade local: o
Testemunho de recebimento de Bob deve referenciar o de gasto de Alice.

CRDTs resolvem consistência eventual sem ordenação global — a mesma garantia
que o protocolo precisa, com fração do custo computacional e sem exigência
de conectividade contínua.

Usar blockchain aqui seria como usar contrato com firma reconhecida em cartório
para combinar de se encontrar no bar. Funcionaria. Seria uma proporção errada
entre instrumento e propósito.
