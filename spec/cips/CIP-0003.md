# CIP-0003: Capsula Agora, Feira e Mutirao

Status: Draft
Data: 2026-04-06
Categoria: Standards Track
Licenca: CC0 1.0 Universal - dominio publico

## Resumo (Abstract)

Define semantica e payloads para tres capsulas de referencia:
Agora (governanca), Feira (mercado local) e Mutirao (trabalho coletivo).
Nao altera o protocolo, apenas padroniza `capsule/invoke` e `capsule/result`.

## Motivacao

Agora, Feira e Mutirao cobrem governanca, trocas e trabalho coletivo
offline-first. Formatos claros permitem interoperabilidade entre apps
e runtime sem impor UX.

## Especificacao

### Envelope de Invocacao

`capsule/invoke` payload (CBOR canonical, chaves tstr):

```
{
  "capsule_id": <bytes32>,  // SHA3-256 do WASM
  "action":     <tstr>,     // nome da acao
  "params":     <bstr>      // CBOR do payload de acao
}

Ordem canonical das chaves (tstr):

- `action` (6) < `params` (6) < `capsule_id` (10)  // desempate: "action" < "params"
```

`capsule/result` payload (CBOR canonical, chaves tstr):

```
{
  "capsule_id": <bytes32>,
  "action":     <tstr>,
  "status":     <uint>,     // 0 = ok, >0 = erro
  "result":     <bstr>      // CBOR do resultado (pode ser vazio)
}

Ordem canonical das chaves (tstr):

- `action` (6) < `result` (6) < `status` (6) < `capsule_id` (10)  // desempate: "action" < "result" < "status"
```

### Capsula Agora

#### Acoes

1) `propose`

```
{
  "proposal_id": <bytes32>,
  "title":       <tstr>,
  "body":        <tstr>,
  "quorum":      <uint64>,
  "expires":     <uint64>,
  "author":      <DID>
}
```

Ordem canonical das chaves (tstr):

- `body` (4) < `title` (5) < `author` (6) < `quorum` (6) < `expires` (7) < `proposal_id` (11)
  - desempate 6: "author" < "quorum"

2) `vote`

```
{
  "proposal_id": <bytes32>,
  "choice":      <uint>,    // 0 = nao, 1 = sim, 2 = abstencao
  "voter":       <DID>
}
```

Ordem canonical das chaves (tstr):

- `voter` (5) < `choice` (6) < `proposal_id` (11)

3) `close`

```
{
  "proposal_id": <bytes32>,
  "timestamp":   <uint64>
}
```

Ordem canonical das chaves (tstr):

- `timestamp` (9) < `proposal_id` (11)

#### Regras

- `proposal_id` MUST ser SHA3-256 do CBOR canonical de `propose` sem o campo `proposal_id`.
- `vote` MUST referenciar um `proposal_id` valido.
- `close` SHOULD ser emitido apos `expires`.
- Voto unico por `voter`: se houver multiplos votos, considerar o de maior `timestamp` do Testemunho; empate desempata por maior `id` lexicografico.
- `quorum` e o numero minimo de votos unicos (sim + nao + abstencao).
- Resultado: `aprovado = (sim > nao) && (quorum_met)`.
- Votos apos `close` ou apos `expires` MUST ser ignorados para o resultado.

#### Resultado de `close` (capsule/result)

O campo `result` SHOULD conter um CBOR map:

```
{
  "proposal_id": <bytes32>,
  "yes":         <uint64>,
  "no":          <uint64>,
  "abstain":     <uint64>,
  "quorum_met":  <uint>,   // 0 = nao, 1 = sim
  "passed":      <uint>    // 0 = nao, 1 = sim
}
```

Ordem canonical das chaves (tstr):

- `no` (2) < `yes` (3) < `passed` (6) < `abstain` (7) < `quorum_met` (10) < `proposal_id` (11)
- Observacao: ordenar por tamanho UTF-8; em empate, lexicografico.

### Capsula Mutirao

#### Acoes

1) `task`

```
{
  "task_id":    <bytes32>,
  "title":      <tstr>,
  "details":    <tstr>,
  "reward":     <uint64>,
  "expires":    <uint64>,
  "creator":    <DID>
}
```

Ordem canonical das chaves (tstr):

- `title` (5) < `reward` (6) < `creator` (7) < `details` (7) < `expires` (7) < `task_id` (7)
  - desempate 7: "creator" < "details" < "expires" < "task_id"

2) `commit`

```
{
  "task_id":    <bytes32>,
  "worker":     <DID>
}
```

Ordem canonical das chaves (tstr):

- `worker` (6) < `task_id` (7)

3) `checkin`

```
{
  "task_id":    <bytes32>,
  "timestamp":  <uint64>
}
```

Ordem canonical das chaves (tstr):

- `task_id` (7) < `timestamp` (9)

4) `complete`

```
{
  "task_id":    <bytes32>,
  "timestamp":  <uint64>
}
```

Ordem canonical das chaves (tstr):

- `task_id` (7) < `timestamp` (9)

5) `reward`

```
{
  "task_id":    <bytes32>,
  "amount":     <uint64>,
  "timestamp":  <uint64>
}
```

Ordem canonical das chaves (tstr):

- `amount` (6) < `task_id` (7) < `timestamp` (9)

#### Regras

- `task_id` MUST ser SHA3-256 do CBOR canonical de `task` sem o campo `task_id`.
- `commit` SHOULD ocorrer antes de `checkin`.
- `reward` SHOULD ser emitido apos `complete`.

## Compatibilidade

Nao ha breaking changes no protocolo.

## Consideracoes de Seguranca

- Recomendado exigir prova de contexto para `offer`, `accept` e `reward`.
- Disputas podem exigir regras adicionais via Capsula Agora.
