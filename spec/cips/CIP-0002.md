# CIP-0002: Commoner Facade (API de No)

Status: Draft
Data: 2026-04-06
Categoria: Standards Track
Licenca: CC0 1.0 Universal - dominio publico

## Resumo (Abstract)

Define um facade de alto nivel para representar um no do Comum
(`Commoner`), unificando emissao, ingestao, validacao e sync de
Testemunhos, sem alterar o protocolo.

Nota terminologica: neste documento, `Commoner` e o facade tecnico que
encapsula um `Node`. O papel social (agente) pode ser chamado de Commoner
na teoria; aqui o foco e a interface de no.

## Motivacao

As implementacoes atuais expõem primitives isoladas (CTE, sync, validacao),
mas nao oferecem um objeto unico para orquestrar um no. Isso cria atrito
na integracao de apps e SDKs. Um facade explicito reduz divergencias e
preserva as garantias normativas.

## Especificacao

### Interface minima

```
Commoner {
  did() -> DID
  clock() -> uint64

  validate(testimony) -> Result
  ingest(testimony) -> Result

  emit(verb, payload, context) -> Testimony

  build_hello(profile) -> HelloPayload
  build_request(clock, limit) -> RequestPayload
  apply_response(payload) -> Result

  encode_cte(payload) -> CTE
  fragment_cte(cte, mtu, frag_id) -> [CTEFragment]
  reassemble(fragments) -> CTE
}
```

### Regras

- O facade MUST preservar as garantias normativas da CIP-0001.
- O facade MUST NOT suprimir erros de validacao.
- Erros MUST distinguir:
  - formato (CBOR/CTE)
  - prova (assinatura/nullifier/context)
  - estado (clock/prev_id/refs)

## Compatibilidade

Nao ha breaking changes no protocolo.

## Consideracoes de Seguranca

- O facade nao deve aceitar Testemunhos invalidos por conveniencia.
- Implementacoes SHOULD expor logs/reason codes para auditoria.
