# CIP-0004: ZK Identity Commitments and Session Proofs

Status: Draft
Data: 2026-04-11
Categoria: Standards Track
Licenca: CC0 1.0 Universal - dominio publico

## Resumo (Abstract)

Define um mecanismo de identidade soberana com provas de conhecimento-zero
compativel com o modelo de `Testimony` da CIP-0001. A extensao usa novos
verbos em `claim.verb`, payloads CBOR canonicos em `claim.payload` e, quando
necessario, reutiliza `proof.zk_proofs` sem alterar a estrutura base do
Testemunho.

O objetivo e permitir que um participante prove controle sobre um commitment
de identidade e gere nullifiers de sessao sem revelar sua chave Falcon-512 nem
atributos pessoais.

Este draft foca no fluxo online minimo. O fluxo offline e tratado como extensao
futura para evitar acoplamento prematuro entre pedido de autenticacao e
registro persistente no DAG.

## Motivacao

O Comum Protocol autentica Testemunhos por assinatura. Isso preserva autoria,
mas tambem revela um identificador criptografico persistente do autor. Em
contextos onde anonimato ou pseudonimato local sao desejados, isso cria um
vinculo rastreavel entre participacao e identidade criptografica.

Este CIP introduz uma camada opcional para:

- registrar commitments de identidade no DAG;
- provar conhecimento do segredo por tras desse commitment sem revelar a chave;
- emitir nullifiers de sessao para prevenir replay;
- permitir onboarding mediado por vouch sem criar autoridade global.

## Especificacao

### Escopo e compatibilidade

- Esta proposta depende da CIP-0001 como base normativa para `Testimony`,
  `claim.verb`, `claim.payload`, `context`, `proof`, suites criptograficas e
  nullifiers do Testemunho.
- Esta proposta NAO introduz campos novos no topo de `Testimony`.
- Esta proposta NAO redefine `proof.nullifiers` da CIP-0001.
- Esta proposta usa HKDF-SHA3-256, em alinhamento com a suite da CIP-0001.

### Primitivos

- KDF: HKDF-SHA3-256
- Hash ZK-friendly: Poseidon sobre BN254
- Perfis de prova suportados: `groth16` e `ultrahonk`

Dominios HKDF:

```
DOMAIN_COMMITMENT = "comum:zk:commitment:v1"
DOMAIN_NULLIFIER  = "comum:zk:nullifier:v1"
```

### Novos verbos

Implementacoes que suportam este CIP MUST reconhecer os seguintes verbos:

- `comum/identity_commitment`
- `comum/auth_nullifier`
- `comum/identity_vouch`

Os verbos `comum/key_rotate` e `comum/revoke` continuam sendo os mecanismos
normativos de rotacao e revogacao ja definidos na base do protocolo.

### Payload: comum/identity_commitment

`claim.payload` MUST conter CBOR canonico equivalente a:

```cddl
IdentityCommitmentPayload = {
  commitment:   bstr .size 32,
  epoch:        uint,
  proof_system: "groth16" / "ultrahonk",
  circuit_id:   bstr .size 32,
  ?supersedes:  bstr .size 32,
}
```

Onde:

```
falcon_pk_hash = HKDF-SHA3-256(ikm=falcon_public_key, info=DOMAIN_COMMITMENT, L=32)
commitment     = Poseidon(falcon_pk_hash, secret_salt)
```

Regras:

- `secret_salt` MUST ser gerado localmente e MUST NOT ser transmitido.
- `supersedes`, quando presente, referencia o `testimony_id` do commitment
  anterior no mesmo contexto.
- O Testemunho que carrega `comum/identity_commitment` MUST seguir a validacao
  normal de assinatura da CIP-0001.

### Payload: comum/auth_nullifier

`claim.payload` MUST conter CBOR canonico equivalente a:

```cddl
AuthNullifierPayload = {
  session_nullifier: bstr .size 32,
  session_nonce:     bstr .size 32,
  commitment_ref:    bstr .size 32,
  resource_id:       bstr .size 32,
  proof_system:      "groth16" / "ultrahonk",
  circuit_id:        bstr .size 32,
}
```

Onde:

```
session_nullifier = Poseidon(secret_salt, session_scope)
session_scope     = session_nonce
```

Perfis futuros MAY derivar `session_scope` a partir de `context_id` e
`resource_id`, mas esta versao fixa o escopo em `session_nonce` para manter o
circuito minimo.

Regras:

- `session_nullifier` e distinto de `proof.nullifiers` da CIP-0001.
- `session_nullifier` previne replay de autenticacao.
- `proof.nullifiers` continua reservado ao mecanismo anti double-spend do
  proprio Testemunho.
- Um no MUST rejeitar ou ignorar `comum/auth_nullifier` cujo
  `session_nullifier` ja exista no estado local.

### Payload: comum/identity_vouch

`claim.payload` MUST conter CBOR canonico equivalente a:

```cddl
IdentityVouchPayload = {
  commitment_ref: bstr .size 32,
  voucher_depth:  uint,
  vouch_nonce:    bstr .size 32,
}
```

Onde:

```
vouch_nonce = HKDF-SHA3-256(
  ikm=voucher_public_key || commitment_ref,
  info=DOMAIN_NULLIFIER,
  L=32
)
```

Regras:

- `vouch_nonce` MUST ser deterministico por `(voucher, commitment_ref)`.
- O mesmo autor MUST NOT emitir mais de um vouch efetivo para o mesmo
  `commitment_ref`.
- Politicas como `M` vouches minimos e `D` profundidade minima sao locais ao
  contexto e MUST NOT ser impostas globalmente por este CIP.

### Prova ZK de sessao

O fluxo online de autenticacao usa uma prova ZK efemera, enviada ao verificador
fora do DAG. O statement minimo e:

> Eu conheco um `secret_salt` tal que
> `Poseidon(HKDF-SHA3-256(falcon_public_key, DOMAIN_COMMITMENT), secret_salt)`
> produz o `commitment` registrado, e `Poseidon(secret_salt, session_nonce)`
> produz o `session_nullifier` apresentado.

Inputs publicos canonicos:

```
index 0: commitment
index 1: session_nonce
index 2: context_id
index 3: session_nullifier
```

Inputs privados:

- `secret_salt`
- `falcon_pk_hash`

### Encapsulamento em proof.zk_proofs

Quando uma prova ZK deste CIP for carregada dentro de um Testemunho,
`proof.zk_proofs` MUST continuar obedecendo o schema base `[* bstr]`.

Cada entrada MUST ser um `bstr` contendo o CBOR canonico de:

```cddl
ZKProofEntry = {
  circuit_id:    bstr .size 32,
  proof_system:  "groth16" / "ultrahonk",
  proof_bytes:   bstr,
  public_inputs: [+ bstr .size 32],
}
```

Este CIP nao exige que `comum/identity_commitment` carregue a prova de sessao.
O uso mais direto e:

- `comum/identity_commitment`: registro persistente no DAG;
- prova ZK de sessao: mensagem efemera entre participante e verificador;
- `comum/auth_nullifier`: recibo persistente anti-replay apos verificacao.

### Fluxo minimo

#### Registro

1. O participante gera `secret_salt` localmente.
2. Calcula `commitment` a partir da chave Falcon e do `secret_salt`.
3. Emite um Testemunho com `claim.verb = "comum/identity_commitment"`.
4. O Testemunho e propagado pelo DAG.
5. Participantes existentes MAY emitir `comum/identity_vouch`.

#### Autenticacao online

1. O verificador emite `session_nonce`.
2. O participante localiza um `commitment` valido no contexto.
3. O participante gera prova ZK local e calcula `session_nullifier`.
4. O participante envia `(proof, public_inputs)` ao verificador.
5. O verificador confirma existencia e validade do commitment.
6. O verificador confirma ausencia previa do `session_nullifier`.
7. O verificador valida a prova.
8. Se tudo for valido, o verificador emite `comum/auth_nullifier` no DAG.

#### Rotacao e revogacao

- Rotacao de commitment usa novo `comum/identity_commitment` com `supersedes`.
- Revogacao usa `comum/revoke` referenciando o `commitment_ref` relevante.

## Compatibilidade

Nao ha breaking change no modelo base de `Testimony`.

Impactos esperados:

- atualizar `spec/registries/verbs.md` com os tres novos verbos;
- atualizar `impl/comum-rs/src/verbs.rs` com novas constantes;
- adicionar validacao de payloads nas implementacoes Rust/JS;
- manter `spec/schemas/testimony.cddl` inalterado se `proof.zk_proofs`
  permanecer `[* bstr]` com semantica definida por CIP.

## Consideracoes de Seguranca

- Vazamento de `secret_salt` compromete a capacidade de provar autenticacao;
  implementacoes SHOULD permitir rotacao rapida de commitment.
- `session_nullifier` MUST ser unico por sessao para prevenir replay.
- Implementacoes SHOULD adicionar pequena variacao temporal antes do envio da
  prova para reduzir fingerprinting por latencia.
- BN254 e Poseidon nao sao primitives pos-quanticas; isso afeta o sistema de
  prova, nao a semantica do DAG.
- Comunidades pequenas ou onboarding imediatamente seguido de autenticacao
  podem reduzir o conjunto de anonimato por correlacao temporal.

## Vetores de Teste

Implementacoes SHOULD fornecer ao menos:

- derivacao de `commitment` com entradas conhecidas;
- derivacao de `session_nullifier` com entradas conhecidas;
- prova valida e prova invalida para o circuito base;
- rejeicao de replay para `session_nullifier` repetido;
- rotacao de commitment com `supersedes`;
- validacao de `ZKProofEntry` serializado em `proof.zk_proofs`.
